## Abstract

Rare disease data portals depend on manual curation to discover and index publicly deposited datasets, a process that cannot keep pace with the rate of data deposition. We describe NADIA (Notable Asset Discovery, Indexing, and Annotation), an autonomous agent that continuously monitors scientific repositories for disease-relevant datasets and provisions them for expert review. NADIA uses a publication-first architecture: it searches PubMed for disease-relevant papers, then resolves linked dataset accessions across fifteen repositories including GEO, SRA, ENA, PRIDE, and Zenodo. The agent, powered by Claude (Anthropic), scores each candidate against domain-specific relevance criteria and populates metadata annotations according to a controlled-vocabulary schema using a four-tier gap-fill strategy that progresses from structured repository metadata to full-text literature extraction. Candidates flagged as `pendingReview` enter a human-in-the-loop workflow where data managers approve, correct, or reject each entry before it is published to the portal. A self-improvement loop incorporates annotation corrections from human reviewers into a persistent skill file that refines future runs. In its first four weeks targeting neurofibromatosis, NADIA indexed over 219 datasets while accumulating 21 annotation rules across two self-improvement cycles. NADIA's configuration is fully domain-agnostic and the architecture is adaptable to any disease-specific portal backed by a structured metadata schema.

## Introduction

Research data portals are essential infrastructure for rare disease communities, where the limited patient population makes cross-laboratory data integration particularly important for achieving statistical power in translational research. The neurofibromatoses (NF1, NF2, and schwannomatosis) are autosomal dominant tumor predisposition syndromes collectively affecting approximately 1 in 3,000 individuals, caused by loss-of-function mutations in *NF1*, *NF2*, *SMARCB1*, or *LZTR1*. The NF Data Portal (nf.synapse.org), maintained by Sage Bionetworks on behalf of the NF research community, aggregates genomic, proteomic, imaging, and clinical datasets from both funded NF research consortia and independent investigators worldwide.

Datasets generated by funded consortia enter the portal through a structured submission process. Datasets from independent investigators, however, have historically been discovered ad hoc, through manual literature searches or community reports. This reactive approach introduces multi-year lags between publication and portal availability and systematically misses datasets deposited without an accompanying paper.

Several recent systems have demonstrated that large language models (LLMs) can substantially reduce the labor of standardizing existing metadata. Combining GPT-4 with structured metadata templates achieves >90% accuracy on controlled-vocabulary annotation tasks for datasets already in GEO or BioSample [1,2], and multi-agent architectures that decompose curation into extraction, normalization, and validation sub-tasks have extended this further [3,4]. These systems primarily address metadata *quality* for datasets already known to the portal.

NADIA addresses a complementary problem: autonomous *discovery* of datasets not yet indexed in a disease-specific portal, followed by structured annotation and human review before publication. This closes the gap between data deposition and portal availability at a scale manual curation cannot sustain.

## System Description

### Architecture

NADIA runs as a daily GitHub Actions workflow that invokes a Claude Code agentic session (Anthropic). Unlike pipelines that call LLM APIs to execute predefined steps, the agent dynamically writes and executes Python scripts in response to what it finds at each stage, adapting its approach based on repository responses, metadata availability, and deduplication outcomes. The agent's behavior is governed by a plain-text instruction set (CLAUDE.md), a runtime configuration file specifying the disease domain, target portal, and controlled vocabularies (`config/settings.yaml`), and a persistent skill file of annotation patterns learned from human corrections.

### Discovery

NADIA uses a publication-first architecture (Figure 1). The primary pass queries PubMed using a structured MeSH plus keyword query (configurable per disease domain), then resolves linked datasets for each returned paper through: NCBI elink (GEO, SRA, dbGaP), the PubMed DataBankList, CrossRef relations API, and the Europe PMC annotations API. Together these channels capture datasets regardless of whether authors submitted accession numbers through their journal's data availability workflow. A secondary pass queries fifteen repositories directly, including Zenodo, Figshare, OSF, PRIDE, MetaboLights, ArrayExpress, EGA, MassIVE, NCI PDC, NCI GDC, Cell Image Library, and the DataCite API (which indexes institutional and national repositories not covered by the others), and skips any result whose PMID was already found in the primary pass.

Accessions returned by NCBI elink require ownership verification: elink frequently returns accessions from papers other than the one being processed. NADIA validates each GEO accession by fetching its associated PubMed IDs and confirming a match before using it.

For repository-direct candidates without a linked publication, NADIA attempts to resolve one via DOI lookup in PubMed, title search, and bioRxiv keyword search before falling back to the repository record itself as the source of authorship metadata.

### Relevance scoring and rejection

For each discovered candidate, NADIA constructs a *publication group* (the paper's title, abstract, and all linked accessions) and reasons directly over it to assign a relevance score between 0.0 and 1.0, assess whether the deposit contains primary data, and determine access type. Candidates below a threshold of 0.70, lacking primary data, or accessible only under embargo are rejected. Explicit rejection rules additionally exclude re-analysis studies (papers reprocessing existing public data without depositing new primary data), summary-only deposits (aggregate statistics files without raw or processed primary data), and stub repositories (records with no files deposited). A safety cap of 50 Synapse write operations per run prevents runaway provisioning during the discovery of large backlogs.

### Deduplication

Before creating any portal record, NADIA checks three sources: its own state table of previously processed accessions; exact accession string matches in the portal's `alternateDataRepository` field; and fuzzy title matching using TF-IDF cosine similarity (threshold ≥ 0.85) combined with Jaccard unigram overlap (threshold ≥ 0.50). Candidates are classified as NEW (create a Synapse project), ADD (add a new accession to an existing project), or SKIP (already fully indexed).

### Annotation

Approved candidates are provisioned as Synapse project entities [Synapse; CITE] with structured annotations drawn from the NF metadata dictionary (github.com/nf-osi/nf-metadata-dictionary), a controlled-vocabulary schema maintained by the NF-OSI curation team. Metadata is populated through a four-tier gap-fill strategy applied in priority order:

1. **Structured repository metadata:** ENA filereport, BioSample XML, GEO sample characteristics
2. **Publication metadata:** PMC full text, supplementary tables, CrossRef funder records
3. **LLM text extraction:** abstract and methods section reasoning for unambiguous values
4. **Data file inspection:** h5ad observation columns, BAM header tags, count matrix headers

File-level annotations are populated per sample, not at study level, using run accessions (SRR, ERR) to map each file back to individual sample metadata. Following annotation, a self-audit step corrects identifiable issues before human review, including resource status annotations erroneously applied to File entities, specimen identifiers duplicated across files from multi-sample studies, and NaN artifacts from metadata serialization.

Each new project is flagged `pendingReview` and a GitHub issue is filed to notify the data management team. Managers may approve the entry, post structured correction requests (`/nadia fix: <description>`) that trigger a targeted annotation update, or reject the entry. Approved entries are provisioned to the live portal by a separate, code-only workflow with no LLM involvement.

### Self-improvement

Each week, NADIA runs a "Dream" cycle: it reviews recently closed GitHub issues, extracts concrete annotation rules from human corrections and confirmed approvals, and adds them to a persistent skill file (`annotation_patterns.yaml`). On subsequent runs, these rules are loaded before any annotation work begins. This mechanism accumulates expert knowledge without model retraining and with compounding returns as the rule set grows.

## Results

### Deployment

We deployed NADIA on the NF Data Portal on March 26, 2026. An initial five-year lookback run recovered the backlog of independently published datasets not yet in the portal; subsequent daily runs operate on a rolling 30-day window. Over the first four weeks of operation, NADIA provisioned over 219 dataset entries for data manager review. Data types included bulk RNA-seq, single-cell RNA-seq, proteomics, epigenomics, imaging, and clinical genomics across GEO, SRA/ENA, Zenodo, PRIDE, EGA, and institutional repositories.

### Self-improvement

Two Dream cycles were completed (on April 2 and April 16, 2026), accumulating 21 concrete annotation rules. Rules span the following categories: investigator attribution (repository submitter ≠ principal investigator), per-sample field population (specimen ID must be unique per file), schema enum handling (empty enum fields must not be set), full-text fallback order (PMC → Unpaywall → Semantic Scholar), assay specificity (single-cell RNA-seq vs. bulk), and disease annotation scoping (germline NF mutation vs. somatic NF loss in general cancer). The most impactful single rule, derived from multiple corrected projects, concerns disease annotation scoping: NADIA initially annotated somatic *NF1* loss in general cancer cell lines as NF1 disease data, conflating it with datasets from patients with NF1. The Dream cycle encoded this distinction explicitly. This misclassification has not recurred in subsequent runs.

## Discussion

NADIA demonstrates that a general-purpose LLM agent operating within a structured human-in-the-loop workflow can systematically reduce the lag between dataset deposition and rare disease portal availability. The publication-first architecture is more precise than repository-direct keyword queries alone: publication records provide the semantic context (title, abstract, author affiliations, funding, and data availability statements) needed to distinguish disease-relevant deposits from coincidental matches.

The human-in-the-loop model is a deliberate design choice. Setting new entries to `pendingReview` rather than directly publishing them ensures that annotation errors, misclassified disease relevance, or inappropriate disease annotations are caught before reaching portal users. This is especially important for a small-disease-community portal where erroneous entries reduce user trust. The `/nadia fix` command provides a lightweight feedback channel that keeps human correction effort low while preserving human authority over final annotations.

NADIA addresses *dataset discovery* rather than metadata harmonization of datasets already known to a portal, a role complementary to systems like MetaMuse [4] and multi-agent GEO curation frameworks [3], which assume the dataset is already identified and focus on improving annotation quality. The two approaches could readily be combined: NADIA handles discovery and initial annotation; a harmonization system provides deeper validation and ontology normalization.

A practical limitation is access to full-text content for subscription-only papers. NADIA falls back through a chain of Unpaywall-indexed preprints, author manuscripts, and Semantic Scholar abstracts before declaring a paper inaccessible; this recovers approximately 50–60% of non-PMC papers but leaves annotation gaps for subscription-only publications. A second limitation is that the relevance threshold of 0.70, while calibrated to minimize false positives, may exclude studies in adjacent disease contexts (e.g., somatic *NF1* loss in sporadic tumors) that some researchers would find valuable. The threshold is configurable in `settings.yaml`, allowing portal operators to tune the precision–recall tradeoff for their community.

NADIA's configuration is fully domain-agnostic: disease vocabulary, search query, target portal, annotation schemas, and controlled vocabularies are all external configuration with no disease-specific logic in the core agent. The same architecture is applicable to any research community maintaining a Synapse-backed data portal with a structured metadata schema. We expect the publication-first discovery approach and the self-improvement loop to generalize across domains.
